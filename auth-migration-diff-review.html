Auth Migration — Diff Review

7-change auth improvement plan · branch main at abe4eb8 · 8 files changed

Executive Summary

The old auth flow was fragile: users had to manually run auth refresh to revive expired sessions, headless Playwright was the only browser path (slow, resource-heavy), and cookie state was thrown away after every login. A failed API call meant silent data loss.

This migration unifies authentication into a single auth login command that tries CDP (Chrome DevTools Protocol) first for instant cookie extraction, then falls back to a persistent headless browser context. Expired sessions now auto-recover inside _request() without user intervention. The Node.js CDP fallback — a source of XSS and thread-safety bugs — has been deleted entirely.

Core insight: Treat auth as a transparent background concern, not a user-facing ceremony. Sessions heal themselves; cookies carry metadata for staleness detection; and one command handles all login paths.

Feature Comparisons

Change 1 (P0): Auto-Refresh on Session Expiry

Before
  • API calls fail silently on session expiry
  • User must manually run auth refresh
  • No retry logic in _request()
After
  • _do_request() extracts retryable HTTP logic
  • _request() catches SessionExpiredError
  • Auto-attempts CDP cookie refresh once
  • Retries the original request after refresh

Change 2 (P0): CDP-First Login

Before
  • Headless Playwright was the only path
  • Slow — launches full browser, navigates pages
  • cmd_auth_login() had no CDP awareness
After
  • cmd_auth_login() tries CDP first
  • Falls back to headless only if CDP fails
  • --headless flag to skip CDP entirely
  • --cdp-only to prevent headless fallback

Before

After

  • Faster subsequent logins
  • --clean flag to wipe state if needed

Code Review

Good

Auto-recovery in _request(). Catching SessionExpiredError and retrying once after CDP refresh eliminates the most common user pain point. The single-retry cap prevents infinite loops.

Good

CDP-first strategy. Trying the fast path (HTTP to localhost:PORT) before spinning up Playwright is the right ordering. Most users have a browser open; CDP takes milliseconds vs. seconds for headless.

Good

Backward-compatible cookie format. load_cookies() handles both old (raw dict) and new ({"cookies":..., "extracted_at":...}) formats. No migration script needed.

Good

Dead code removal. cmd_auth_refresh and _cdp_get_cookies_node() deleted, not deprecated-in-place. Clean cut.

Ugly

warn_if_cookies_stale() was dead code. Defined in config.py but never called anywhere until this migration wired it into cmd_auth_status. The function existed for weeks without being invoked — a reminder that adding a helper without wiring it is half-done work.

Ugly

--cdp-port not on auth login. Users must set LIGHTHOUSE_CDP_PORT env var or use auth refresh --cdp-port for non-default ports. This is an intentional design choice (not a bug) to keep auth login simple, but may surprise power users.

Question

What if CDP port is stale? The auto-refresh in _request() uses the configured CDP port. If the browser was restarted on a different debug port, the refresh will fail silently and the user sees the session-expired error. Should there be a port-discovery mechanism?

Decision Log

Auto-refresh inside _request() High Confidence

Decision: Catch SessionExpiredError at the HTTP layer and attempt a single CDP cookie refresh + retry before failing.

Rationale: Session expiry is the #1 support request. Users don't understand why commands fail. Making recovery automatic eliminates an entire class of user confusion. Single retry prevents infinite loops.

Alternatives: (1) Separate auth refresh command — rejected because users forget to run it. (2) Background refresh timer — rejected as over-engineering for a CLI tool.

CDP-first, headless-fallback ordering High Confidence

Decision: Always try Chrome DevTools Protocol before launching a headless browser.

Rationale: CDP is instant (HTTP request to existing browser). Headless Playwright takes seconds and eats RAM. Most users have a browser open during CLI usage.

Alternatives: Headless-only was the previous default. CDP-only would fail for users without an open browser.

warn_if_cookies_stale() only on valid sessions High Confidence

Decision: Warn about cookie age only inside the if valid: block of cmd_auth_status.

Rationale: Warning on an already-expired session is noise — the user already knows they need to re-auth. Staleness warnings are only useful when the session is technically valid but aging toward expiry.

No --cdp-port on auth login High Confidence

Decision: Keep --cdp-port off auth login. Users use auth refresh --cdp-port or LIGHTHOUSE_CDP_PORT env var.

Rationale: auth login is for the common case — auto-discover or headless. Port configuration is a power-user concern, better served by env var or auth refresh which is the targeted refresh command.

Delete Node.js CDP fallback entirely High Confidence

Decision: Remove _cdp_get_cookies_node() rather than fix it.

Rationale: The Node.js path had XSS and thread-safety bugs. Python HTTP to localhost:PORT covers all real use cases. Maintaining two CDP paths is unnecessary complexity.

Re-entry Context

Key Invariants

  • load_cookies() must handle both old (raw dict) and new ({"cookies":..., "extracted_at":...}) formats. Tests cover both paths.
  • HeadlessAuthenticator must be used as a context manager. If someone calls methods without with, the browser won't close.
  • Auto-refresh in _request() retries exactly once. Changing this requires updating the retry-cap logic.
  • BROWSER_STATE_DIR must exist before launch_persistent_context(). --clean wipes it.

Non-obvious Coupling

  • api.py imports from auth.py for auto-refresh. Changing the CDP function signature in auth.py breaks the retry in api.py.
  • config.py cookie format is coupled to auth.py save and load. Adding fields to the metadata wrapper requires updating both.
  • warn_if_cookies_stale() reads extracted_at from cookie metadata. If the cookie file is in old format (no timestamp), the function returns None silently.

Gotchas

  • --clean flag wipes BROWSER_STATE_DIR which is also used for headless. Running auth login --clean followed by auth login will be slower (no cached state).
  • CDP port discovery happens at login time. If the browser restarts on a different port, existing cookies are still valid until expiry — but auto-refresh will fail to reach CDP.
  • asyncio.run() creates a new event loop each call. If HeadlessAuthenticator methods are called from an existing event loop (e.g., in a future async CLI), this will crash. The context manager pattern mitigates this by scoping the loop.
